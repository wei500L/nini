# Newsroom 三层记忆设计

Newsroom 不把“记忆”笼统地等同于一段越来越长的 prompt，而是按生命周期和可信度拆成三层：短期记忆（对话历史）、会话内状态（`fact_state`）和跨会话记忆（`profile`）。三层分别回答“刚才说了什么”“这场采访推进到了哪里”“这个学生长期需要练什么”，不能互相替代。

| 层级 | 生命周期 | 典型内容 | 谁负责更新 | 主要用途 |
| --- | --- | --- | --- | --- |
| 短期记忆：对话历史 | 单场、随上下文窗口滚动 | 最近几轮主持人问题、嘉宾回答、导播提示 | 编排器按 turn 追加 | 让嘉宾承接上文，让导播判断是否追问、是否刚提示过 |
| 会话内状态：`fact_state` | 单场开始到复盘结束 | 每条 fact 的 `guard_current`、`consecutive_probes`、`revealed` | 纯代码状态机 | 决定嘉宾继续回避、露破绽、部分松口还是完整揭示，并保证同一场内事实进度唯一 |
| 跨会话记忆：`profile` | 多场长期保留 | `metrics_history`、`chronic_weaknesses`、`filler_blacklist`、各 persona 最高分 | 纯代码在每场结束后汇总 | 改变下一场导播和评委的关注重点，生成成长曲线与跨场对比 |

## 1. 短期记忆：对话历史

对话历史保存逐轮发生的可见事件。它适合处理语义承接，例如嘉宾上一句是否改口、主持人这一问是否接住了关键词、导播最近两轮是否已经说过话。为控制延迟和噪声，实时导播只读取最近四轮，而完整逐字稿在复盘阶段交给评委。

这层记忆是“证据流”，不是权威状态。某句话出现过不等于某条料已经解锁；嘉宾说“我已经解释过了”也不能自动把 fact 标成 `revealed=full`。是否解锁必须由下一层的确定性状态机判断。

## 2. 会话内状态：`fact_state`

每场开始时，系统根据 dossier 为每条 fact 建立一行 `fact_state`：

- `guard_current`：当前防线强度；
- `consecutive_probes`：主持人是否在同一条线上连续施压；
- `revealed`：`hidden / partial / full` 三态。

主持人每问一轮，代码先根据压力、目标 fact、人设响应方式和上一轮状态计算新状态，再让嘉宾依据这个结果生成回答。数据库中的 `fact_state` 是这场采访唯一可信的进度源；导播、嘉宾、速记器和复盘都读取同一份状态。

### 为什么不能把 `fact_state` 塞进 LLM 上下文，靠模型自己记

第一，LLM 输出具有概率性。相同历史重试一次，模型可能把同一条 fact 一次判断为 partial、另一次判断为 full，状态会漂移。

第二，上下文不是事务。嘉宾和导播并发调用时，各自可能根据不同版本的“记忆”继续推理，造成前后矛盾；请求重试还可能把一次状态变化重复执行。数据库状态机则能在一次持久化操作中统一写入 turn 和 fact_state。

第三，上下文会被裁剪。访谈变长后，早期轮次可能离开窗口；如果状态只隐含在旧对话中，模型就可能忘记已经松口的事实，重新否认，或把尚未解锁的事实当成已公开内容。

第四，自然语言无法提供强约束。`guard_current` 必须保持 0–5，`revealed` 只能单向推进，连续追问计数必须按规则重置；这些不变量用 Pydantic、数据库字段和纯代码可以验证，用 prompt 只能“要求”，不能保证。

第五，课程报告需要可审计证据。显式 `fact_state` 可以逐轮回放“哪一问让 guard 降低、何时从 partial 变 full”；若状态只存在模型的隐式理解里，就无法复现，也无法证明信息收获得分没有被 LLM 篡改。

因此，LLM 可以读取 `fact_state` 来决定措辞，但不能拥有或修改它。模型负责语言，代码负责真相。

## 3. 跨会话记忆：`profile`

每场复盘完成后，系统把本场 Metrics 快照追加到 `metrics_history`，并用纯代码更新三类长期结论：

- 最近连续三场都触发阈值的项目进入 `chronic_weaknesses`；不足三场不下“老毛病”结论；
- 全部历史的口头禅频率累计排序，TOP5 写入 `filler_blacklist`；
- 每种 persona 保存个人最高分，未训练过的 persona 明确记为 `null`。

`metrics_history` 同时保留本场五维分数和 `top3_advice`，所以复盘页可以计算“上一场 → 本场”的分差，成长曲线也不需要重新调用 LLM。

## 4. 记忆必须反哺决策

本设计最关键的证据不是“数据库里多了一张表”，而是同一个学生在下一场会得到不同的 agent 行为。

会话创建时，编排器读取 profile。实时导播 prompt 会收到 `chronic_weaknesses`：如果学生连续三场都有“问题太长”和“不追问”，导播就优先盯住这两项，一看到苗头立即提醒，与老毛病无关的次要问题可以放过。这个优先级仍受三轮节流、正确追问时静默和防泄题规则约束。

复盘时，评委 prompt 会收到上一场的 `top3_advice`，逐条检查本场是否落实；确有改善时，必须结合本场 Metrics 或逐字稿 evidence 明确表扬，未改善则如实指出。复盘页再用代码展示各维度相对上一场的上升、下降或持平箭头。

完整闭环是：

```text
本场 turns + fact_state
        ↓ 纯代码统计
      Metrics
        ↓ 评委复盘
五维分数 + top3_advice
        ↓ 原子更新
      profile
        ↓ 下一场创建时读取
导播关注重点 + 评委跨场检查 + 页面成长对比
```

这形成了可检验的“记忆影响决策”：只要固定当前对话、替换学生 profile 中的慢性弱点，导播 prompt 和提醒优先级就会随之变化；固定本场逐字稿、替换上一场建议，评委的跨场检查目标也会变化。记忆不只是存着好看，而是进入了下一次决策的因果链。

## 5. 边界与一致性

三层都由编排器串联，但权限不同：对话历史可被各 agent 阅读；`fact_state` 只允许代码更新，且导播不得泄露未揭示 fact 内容；`profile` 只保存训练表现，不保存隐藏 dossier。报告写入与 profile 更新放在同一个数据库事务中，避免出现“页面已有复盘但长期记忆漏记一场”的不一致。

这种拆分让系统同时获得自然语言的灵活性和状态机的确定性：短期记忆保证连贯，`fact_state` 保证单场事实一致，`profile` 保证长期教学策略会随学生成长而变化。
